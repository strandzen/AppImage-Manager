# Contributing to AppImage Manager

## Building

### Dependencies (Arch-based)

```bash
sudo pacman -S base-devel cmake extra-cmake-modules ninja \
    qt6-base qt6-declarative \
    kcoreaddons ki18n kio kiconthemes knotifications kcrash kdbusaddons kirigami \
    libcanberra
```

### Build and install

```bash
cmake --preset dev
cmake --build --preset dev
sudo cmake --install build/dev
```

To reload Dolphin after install without a full logout:

```bash
kquitapp6 dolphin && dolphin &
```

### Clean rebuild

```bash
rm -rf build/ && cmake --preset dev && cmake --build --preset dev
```

## Architecture

| Target | Type | Description |
|--------|------|-------------|
| `appimagemanager_core` | Object library | Pure C++, no QML. `AppImageReader`, `AppImageManager`, `AppImageInfo`, `AppSettings` |
| `appimagemanager_qml` | QML module | GUI backend classes + QML files (all windows and dialogs) |
| `appimagemanager` (plugin) | `KAbstractFileItemActionPlugin` | Dolphin right-click integration |
| `appimagemanager` (binary) | Executable | Standalone launcher opened by the Dolphin service menu |

**Two entry points, one codebase.** The Dolphin plugin invokes `AppImageWindow::open(path)`. The binary does the same. Both produce identical UI.

**Data flow:**
```
AppImageWindow::open(path)
  └─ AppImageBackend(path) — emits metadataLoadedChanged
       └─ QtConcurrent::run(AppImageReader(path).read())   ← worker thread
            └─ AppImageInfo (value struct)
       └─ onMetadataReady → updates Q_PROPERTY → QML binds
```

**Dashboard entry point:** `DashboardWindow::open()` is a singleton that creates its own `QQmlApplicationEngine`, `AppImageListModel`, and `AppImageSortFilterModel`. It is opened by `appimagemanager --dashboard`.

## Project Layout

```
src/
  core/           Pure C++ logic (no Qt Quick dependency)
  gui/            Backend classes exposed to QML + window hosts
qml/              All .qml files (registered as QRC resources)
po/               gettext translations; .pot generated by cmake
data/             Desktop entry, service menu, notification, icon
```

## Branching and Commits

- **`main`** is always releasable. Direct commits are fine for small fixes.
- For larger features, open a short-lived feature branch and merge via PR.
- Commit messages follow [Conventional Commits](https://www.conventionalcommits.org/):

  ```
  feat: add storage usage dialog
  fix: center icons correctly when app is installed
  refactor: extract createBackend() helper in DashboardWindow
  docs: update README build instructions
  chore: add CMAKE_EXPORT_COMPILE_COMMANDS to dev preset
  ```

  Types: `feat`, `fix`, `refactor`, `docs`, `chore`, `style`, `test`

## Code Style

- C++20, Qt6, KDE Frameworks 6 conventions.
- Blocking I/O (`AppImageReader`, corpse scan) always runs via `QtConcurrent::run`.
- User file deletion goes through `KIO::trash()`, never `QFile::remove`.
- QML context properties (not creatable types) expose C++ to QML.
- No raw `new` without a parent or immediate `unique_ptr`/`QScopedPointer`.

## Translations

Strings are wrapped in `i18n()` (QML) or `i18n()` / `tr()` (C++, prefer KI18n).
The `.pot` template is regenerated by CMake (`ki18n_install`). To add a language,
copy `po/appimagemanager.pot` to `po/<lang>/appimagemanager.po` and translate.
