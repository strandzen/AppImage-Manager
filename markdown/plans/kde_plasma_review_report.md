# KDE Plasma 6 Integration Review — AppImage Manager

This document presents a technical analysis of the **AppImage Manager** project from the perspective of a KDE Plasma developer, reviewing its readiness for official inclusion in a future release of KDE Plasma 6.X.

---

## 1. What Needs to Be Improved

### Security & Integrity Check in HTTP Downloader
* **Issue:** In `AppImageUpdateManager::startFullHttpDownload` (the fallback downloader for when `zsync2` is not installed), the downloaded AppImage binary is swapped directly into place and marked executable without any hash verification.
* **Impact:** A corrupted download or a Man-in-the-Middle (MitM) tampered file could be executed automatically, posing a significant security risk.
* **Improvement:** Parse the `SHA-1` header from the `.zsync` metadata file during fallback download and verify the downloaded binary's SHA-1 checksum before swapping.

### GitHub API Rate Limiting & Serialization
* **Issue:** `UpdateDaemon::checkUpdates()` checks all AppImages concurrently, making individual HTTP requests to `api.github.com/repos/.../releases/latest` in parallel.
* **Impact:** Users with more than a few AppImages will quickly hit GitHub's unauthenticated API rate limit (60 requests/hour), causing update checks to fail.
* **Improvement:** Queue and serialize update checks instead of executing them all at once. Introduce a retry mechanism or use the Atom feed more eagerly.

### Exec Line Argument Integration (`%U` / `%F`)
* **Issue:** In `AppImageManager::createDesktopLink`, the launcher's `Exec` field is generated by quoting the path and appending custom args, but completely strips XDG placeholders like `%U` or `%F`.
* **Impact:** App launcher search results and Dolphin "Open With" associations will launch the AppImage, but will fail to pass the target file path, rendering file associations useless.
* **Improvement:** Detect and append appropriate XDG command placeholders (`%U` or `%F`) based on the AppImage's internal desktop metadata.

### Background Daemon Concurrency & Thread Spikes
* **Issue:** `UpdateDaemon::checkUpdates` maps files via `QtConcurrent::mapped`, which can spawn up to `QThread::idealThreadCount()` threads on the global thread pool.
* **Impact:** On boot or schedule wakeups, the daemon could cause severe I/O thrashing and CPU spikes on high-core systems reading multi-gigabyte SquashFS images.
* **Improvement:** Use a dedicated, low-concurrency thread pool (similar to the model's `m_readerPool` bounded to 4 threads) to pace SquashFS operations.

## 2. What Needs to Be Removed

### Untrusted Bootstrapping of Third-Party Scripting
* **Issue:** In `AMStoreModel::installApp`, if `appman` or `am` is missing, the code downloads a third-party installation script via `curl` and executes it directly:
  ```cpp
  m_process->start(QStringLiteral("bash"), {QStringLiteral("-c"),
      QStringLiteral("mkdir -p ~/.local/bin && curl -fsSL "
                     "https://raw.githubusercontent.com/ivan-hc/AM/main/appman "
                     "-o ~/.local/bin/appman && chmod +x ~/.local/bin/appman")});
  ```
* **Impact:** This violates core safety guidelines for desktop environments. Executing unverified shell scripts downloaded from the web is unacceptable for system-level inclusion.
* **Removal:** Remove the bootstrap script execution entirely. AppManager should require `appman`/`am` as a packaged system package dependency or fail gracefully with a message indicating the package must be installed.

### QtWidgets Dependency from QML Module
* **Issue:** `AppSettings::openFolderPicker` creates a `QFileDialog` instance, dragging a dependency on the `QtWidgets` module into `appimagemanager_qml`.
* **Impact:** Increases memory usage and blends widget dialog styles with the Kirigami environment.
* **Removal:** Remove `QFileDialog` and the QtWidgets linkage in `appimagemanager_qml`. Replace it with `QtQuick.Dialogs` `FolderDialog` in the QML files (`SettingsPage.qml` / `SettingsDialog.qml`).

## 3. What Needs to Be Added

### Native Download Integration in Discover Store Page
* **Issue:** For applications on the Discover page that do not have an AM installation script, the user is redirected to their web browser via `Qt.openUrlExternally(downloadUrl)`.
* **Impact:** Breaking the in-app installation flow is a poor user experience.
* **Addition:** Implement in-app downloading and auto-integration for Hub-only packages using the backend HTTP download engine.

### Sandboxing Support
* **Issue:** Running AppImages via direct shell execution has security implications, as they can access arbitrary home files.
* **Addition:** Add option settings to run AppImages under a sandbox wrapper like `bubblewrap` or `firejail`, enabling secure isolation.

### Accessibility (a11y) Declarations
* **Issue:** QML layouts do not include accessible name or role declarations.
* **Addition:** Add `Accessible` properties (`Accessible.role`, `Accessible.name`, `Accessible.description`) to all custom lists, buttons, and visual cards to comply with KDE's accessibility standards.

### Standards-Compliant Icon Theme Placement
* **Issue:** Extracted icons are stored in a custom `~/.local/share/icons/AppImages` folder.
* **Addition:** Copy icons to the standard XDG path `~/.local/share/icons/hicolor/scalable/apps/` (or standard raster sizes) and register them properly with the icon theme engine.
